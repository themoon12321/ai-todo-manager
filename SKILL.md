## 时间表达解析

| 用户说 | 解析 |
|--------|------|
| "明天" / "明天上午/下午/晚上" | 当前日期+1，描述中保留时段信息 |
| "后天" | 当前日期+2 |
| "下周一" ~ "下周日" | 下周对应日期 |
| "下个月" | 下月同日 |
| "月底" | 当月最后一天 |
| "EOD" / "今天" | 今天 |
| "next Monday" | 下周一 |
| "by Friday" | 本周五 |

规则：所有相对时间换算为实际日期后填入 deadline，描述中不保留"明天""后天"等相对词汇，改为具体描述。

## 核心流程

### 新增任务

1. 拉取活跃任务表，确定下个可用 ID
2. 解析用户输入：标题、截止日期、分类、标签、语气线索
3. 按 rubrics 多维度打分，确定优先级
4. 判断是否属于某长任务（自然匹配长任务表中的名称）
5. 生成完成建议（见完成建议规则）
6. 写入飞书活跃任务表
7. 触发长任务状态衍生
8. 回复用户

### 完成任务

**方式一：用户主动告知**
1. 定位任务（按 ID 或模糊标题匹配）
2. 记录完成时间 = 今天，清除优先级字段
3. 写入飞书
4. 告知用户

**方式二：用户在飞书手动打勾**
- （无 AI 参与）
- 下次查询/操作时，AI 扫描到状态变更：
  1. 读取飞书系统字段"最后更新时间"，提取实际日期填入"完成时间"
  2. 如果读不到，使用截止日期的次日作为归档依据

### 归档逻辑

**每次操作活跃任务表时顺手检查：**
- 活跃任务表中 `完成时间 < today` 的记录 → 移入归档表，从活跃表删除
- 晚报生成前额外检查一次，确保当日完成记录不丢失

### 查询任务

1. 实时拉取飞书活跃任务表（不依赖缓存）
2. 按用户要求过滤、排序
3. 回复
4. 降级策略：飞书拉取失败 → 读 `tasks_cache.json`，回复时注明

## 优先级多维度打分 Rubrics

### 打分维度

| 维度 | 权重 | 算法 |
|------|------|------|
| **时间紧迫度** | 40% | 今天/已过期→4分，1-2天→3分，3-7天→2分，>7天→1分，无截止→0分 |
| **用户语气** | 20% | "紧急/加急/赶紧"→+2分，"重要/尽快"→+1分，无特殊→0分 |
| **领域权重** | 20% | 学业/项目→基础2分，班务→基础1分，生活→基础0分 |
| **沉寂提权** | 10% | 待办状态每挂3天→+1分，上限3分 |
| **依赖紧密度** | 10% | 有前置任务未完成→+2分，无→0分 |

### 最终等级

| 总分(归一至0-10) | 等级 | 飞书标签样式 |
|-----------------|------|-------------|
| 8-10 | P0 🔥 | 深红色标签 |
| 6-7 | P1 🔴 | 红色标签 |
| 4-5 | P2 🟡 | 黄色标签 |
| 2-3 | P3 🟢 | 青色标签 |
| 0-1 | P4 ⚪ | 灰色标签 |

### 边界值防御（无截止日期任务）

- 时间紧迫度固定计 0 分
- 最终由其余维度重新归一计算
- 默认映射为 P3 或 P4，除非用户语气极度强烈

### 动态调整规则

- **触发时机**：每次新增/更新任务 + 每次操作顺手检查 + 早报前扫描
- **不自动降级**：防止临近任务因语气误判被意外忽略
- **唯一降级特例**：检测到用户主动**推迟截止日期**（新 deadline > 旧 deadline）时，按新日期重新计算
- **用户手动改过优先级的任务不覆盖**（通过 priority_source 字段判断：值为 user 时跳过）

## 长任务状态衍生规则

```
每次操作后，遍历长任务表：
  子任务 > 0 → 状态 = "进行中"
  子任务 = 0 → 状态 = "未开始"
  用户明确说"结项" → 状态 = "已结项"（唯一途径，AI 不自动结项）
  已结项后用户要重开 → 用户手动改或对话说"重开"
```

统计子任务时同时查询活跃任务表和归档表（归档的子任务也算）。

## 完成建议规则

每次新增任务时生成建议，写入"完成建议"字段：

| 维度 | 说明 | 示例 |
|------|------|------|
| 执行方法 | 如何完成的具体步骤 | "按目的→原理→装置→数据→结论顺序写" |
| 时间分配 | 分几段、每段多久 | "分3段：0.5h+1.5h+1h" |
| 优先事项 | 先做什么再做什么 | "先处理数据再写小结" |
| 参考借鉴 | 可参考已完成的任务 | "结构可参考T-001" |
| 前置准备 | 开始前需要什么 | "先复习实验原理" |

规则：
- 同类型任务可套用框架
- 按剩余时间拆解节奏
- 进行中的任务侧重收尾
- 新增时预估>2h的任务拆成2-3个隐含步骤

## lark-cli 操作示例

```bash
# 路径
${LARK_CLI_PATH}

# 拉取活跃任务表
${LARK_CLI_PATH} base +record-list --base-token ${FEISHU_BASE_TOKEN} --table-id ${FEISHU_ACTIVE_TABLE}

# 新增任务
${LARK_CLI_PATH} base +record-upsert --base-token ${FEISHU_BASE_TOKEN} --table-id ${FEISHU_ACTIVE_TABLE} --json '{"任务标题":"xxx"}'

# 更新任务
${LARK_CLI_PATH} base +record-upsert --base-token ${FEISHU_BASE_TOKEN} --table-id ${FEISHU_ACTIVE_TABLE} --record-id ${RECORD_ID} --json '{"状态":"已完成"}'
```

所有字段映射在 `.env` 中引用。实际操作时优先使用 lark-cli 的字段名方式写入。
